# 주문/결제/신청/마이페이지 정합성 개선 기록

> On-Race 팀 프로젝트에서 제가 담당한 주문/결제 정합성 개선 작업을 정리한 문서입니다.  
> 이 문서는 `Order.entryId`, signed prepare token, confirm API, 마이페이지 상태 노출 정책을 중심으로 설명합니다.

---

## 1. 왜 이 작업이 필요했는가

On-Race는 사용자가 마라톤 참가 신청을 하고, 결제한 뒤, 마이페이지에서 신청/주문 상태를 확인하는 서비스입니다.

사용자 입장에서는 아래 흐름이 하나로 보입니다.

```text
마라톤 신청 → 주문 생성 → 결제 진행 → 마이페이지에서 상태 확인
```

하지만 백엔드 내부에서는 이 흐름이 여러 도메인으로 나뉘어 있습니다.

```text
Entry          : 참가 신청
Order          : 주문
Checkout       : 결제 요청
MyPage         : 사용자 화면에 보여줄 신청/주문 상태
```

각 도메인이 서로 다른 기준으로 상태를 처리하면 사용자에게 보이는 정보가 어긋날 수 있습니다.

예를 들어:

```text
신청내역에는 참가 가능한 것처럼 보이는데
주문내역에는 결제 실패로 보이는 상황
```

또는:

```text
이미 만료된 주문인데
신청내역에는 다시 신청 가능한 상태처럼 보이는 상황
```

이런 문제를 줄이기 위해 주문, 신청, 결제, 마이페이지가 같은 기준으로 연결되도록 백엔드 흐름을 정리했습니다.

---

## 2. 문제 상황

### 2-1. 결제 준비와 실제 checkout 단계가 분리되어 있었다

결제 흐름은 보통 한 번의 요청으로 끝나지 않습니다.

On-Race에서도 결제 전에 사용자가 신청한 이벤트, 코스, 페이스, 가격 등을 확인하는 단계가 있고, 이후 실제 checkout 요청이 이어집니다.

문제는 이 두 단계가 분리되어 있을 때입니다.

```text
1. checkout-info 단계에서 결제 준비 정보를 만든다.
2. 사용자가 checkout 요청을 보낸다.
3. 서버는 checkout 요청이 올바른 준비 정보에서 출발했는지 확인해야 한다.
```

이때 검증이 약하면 다음 문제가 생길 수 있습니다.

```text
- 사용자가 신청한 이벤트와 결제 요청의 이벤트가 달라질 수 있음
- 사용자가 신청한 코스/페이스와 결제 요청 정보가 달라질 수 있음
- 이미 사용한 결제 준비 정보가 다시 사용될 수 있음
- 주문이 만들어진 뒤 어떤 신청에서 출발한 주문인지 추적하기 어려움
```

---

### 2-2. 주문과 신청의 연결 기준이 필요했다

주문이 생성된 뒤에도 이 주문이 어떤 신청에서 만들어졌는지 알아야 합니다.

그래야 마이페이지에서 사용자가 보는 상태를 안정적으로 정리할 수 있습니다.

```text
Entry 1개 → 해당 신청에서 만들어진 Order
```

이 연결이 약하면 다음 문제가 생길 수 있습니다.

```text
- 주문은 있는데 어떤 신청과 연결되는지 추적하기 어려움
- 마이페이지에서 신청 상태와 주문 상태를 함께 판단하기 어려움
- 결제 실패/만료/취소 상태를 화면에 어떻게 보여줄지 기준이 흔들림
```

---

### 2-3. 마이페이지는 서버 상태를 그대로 보여주면 안 됐다

마이페이지는 사용자가 직접 보는 화면입니다.

따라서 서버 내부 상태를 그대로 보여주는 것보다, 사용자가 이해할 수 있는 기준으로 상태를 정리해야 합니다.

예를 들어 주문 상태에는 다음과 같은 값이 있을 수 있습니다.

```text
PENDING
PAID
CANCELLED
FAILED
EXPIRED
```

하지만 사용자에게는 이 상태들이 각각 어떤 의미인지 명확해야 합니다.

특히 `FAILED`, `EXPIRED` 같은 상태가 신청내역에 다시 노출되면 사용자가 실제 참가 상태를 오해할 수 있습니다.

---

## 3. 해결 방향

이 작업의 핵심 방향은 아래 3가지였습니다.

```text
1. 주문이 어떤 신청에서 만들어졌는지 연결한다.
2. checkout 요청이 올바른 결제 준비 정보에서 출발했는지 검증한다.
3. 마이페이지에 보여줄 상태 기준을 정리한다.
```

---

## 4. 구현 내용

### 4-1. `Order.entryId` 저장

주문이 생성될 때 해당 주문이 어떤 신청에서 만들어졌는지 저장하도록 `Order.entryId`를 추가했습니다.

#### 목적

```text
주문과 신청을 직접 연결하기 위해
```

#### 기대 효과

```text
- 주문이 어떤 신청에서 만들어졌는지 추적 가능
- 마이페이지에서 신청/주문 상태를 함께 판단 가능
- 결제 완료, 실패, 만료 상태를 신청 상태와 연결해 해석 가능
```

#### 쉽게 설명하면

`Order.entryId`는 주문과 신청을 이어주는 연결 번호입니다.

```text
신청서 번호(entryId)를 주문에도 저장해두면
나중에 이 주문이 어떤 신청에서 나온 주문인지 바로 알 수 있다.
```

---

### 4-2. signed prepare token 검증

결제 준비 단계에서 만든 정보가 실제 checkout 단계까지 바뀌지 않았는지 검증하기 위해 signed prepare token 검증을 추가했습니다.

#### 목적

```text
checkout 요청 변조와 재사용을 줄이기 위해
```

#### 검증해야 하는 정보 예시

```text
- 사용자
- 이벤트
- 코스
- 페이스
- 신청 정보
- 결제 준비 정보
```

#### 기대 효과

```text
- checkout-info 단계와 checkout 단계 사이의 정보 불일치 방지
- 이미 사용한 결제 준비 정보 재사용 방지
- 사용자가 신청한 정보와 다른 정보로 결제되는 문제 감소
```

#### 쉽게 설명하면

signed prepare token은 결제 전에 발급한 “검증표”에 가깝습니다.

```text
결제 전에 만든 신청 정보가
결제 순간에도 같은 정보인지 확인하는 도장
```

예를 들어 A 코스를 신청했는데 checkout 요청에서 B 코스로 바뀌면 안 됩니다.

signed prepare token 검증은 이런 상황을 막기 위한 방어 장치입니다.

---

### 4-3. `POST /orders/{orderNumber}/confirm` API 추가

결제 완료 이후 주문 상태를 확정하기 위한 confirm API를 추가했습니다.

#### 목적

```text
결제 완료 상태를 서버에서 명확하게 확정하기 위해
```

#### 기대 효과

```text
- 주문 상태 전이 기준 명확화
- 결제 완료 후 주문 상태를 안정적으로 PAID 처리
- 이미 PAID 상태인 주문에 대한 중복 처리 기준 마련
```

#### 쉽게 설명하면

checkout은 결제 요청이고, confirm은 서버가 “이 주문은 결제 완료로 확정해도 된다”고 정리하는 단계입니다.

```text
checkout  : 결제를 진행하는 요청
confirm   : 결제 완료를 서버 상태에 반영하는 요청
```

---

### 4-4. 마이페이지 상태 노출 정책 정리

마이페이지에서 신청내역과 주문내역이 서로 다르게 보이지 않도록 상태 노출 기준을 정리했습니다.

#### 다룬 상태

```text
PENDING
PAID
CANCELLED
FAILED
EXPIRED
```

#### 정리한 방향

```text
- 주문이 생성된 신청은 신청내역에서 중복 노출되지 않도록 처리
- FAILED / EXPIRED 주문이 신청내역에 다시 보이지 않도록 처리
- 취소 계열 상태 문구를 사용자에게 이해 가능한 형태로 통일
```

#### 기대 효과

```text
- 사용자가 실제 참가 상태를 오해할 가능성 감소
- 신청내역과 주문내역의 의미 충돌 감소
- 마이페이지 화면의 신뢰도 향상
```

## 5. 전체 흐름

아래는 개선 후 의도한 흐름입니다.

```text
[1] 사용자가 마라톤 참가 신청
        ↓
[2] 신청 정보 생성 Entry
        ↓
[3] 결제 준비 정보 생성 checkout-info
        ↓
[4] signed prepare token 발급
        ↓
[5] checkout 요청 시 token 검증
        ↓
[6] 주문 생성 Order
        ↓
[7] Order.entryId 저장
        ↓
[8] 결제 완료 confirm API 호출
        ↓
[9] 주문 상태 확정
        ↓
[10] 마이페이지에서 신청/주문 상태를 일관되게 표시
```

## 8. 구현하면서 중요하게 본 기준

### 8-1. 기능 하나가 아니라 사용자 흐름으로 보기

처음에는 주문, 결제, 마이페이지가 각각 다른 기능처럼 보일 수 있습니다.

하지만 사용자는 이것을 하나의 흐름으로 경험합니다.

```text
신청했는가?
결제했는가?
내 마이페이지에서 어떤 상태로 보이는가?
```

그래서 API 하나를 만드는 것보다 흐름 전체가 같은 기준으로 동작하는지가 중요했습니다.

---

### 8-2. 서버 내부 상태와 사용자 표시 상태를 구분하기

서버 내부에서는 다양한 상태가 필요합니다.

하지만 사용자에게는 너무 많은 기술적 상태를 그대로 보여주면 혼란을 줄 수 있습니다.

따라서 내부 상태를 기준으로 삼되, 마이페이지에서는 사용자가 이해할 수 있는 표시 정책이 필요했습니다.

---

### 8-3. 운영에서 확인 가능한 구조 만들기

결제와 주문은 문제가 생겼을 때 추적이 가능해야 합니다.

따라서 주문과 신청 연결값을 저장하고, 운영 SQL과 런북을 함께 정리해 문제 발생 시 확인할 수 있도록 했습니다.

---

## 9. 배운 점

### 9-1. 결제 기능은 단순 API 호출이 아니다

결제는 외부 결제 요청만 보내면 끝나는 기능이 아니었습니다.

신청 정보, 주문 정보, 결제 상태, 마이페이지 노출 기준이 모두 연결되어야 했습니다.

이번 작업을 통해 결제 기능은 검증, 상태 전이, 사용자 화면 표시까지 함께 설계해야 한다는 점을 배웠습니다.

---

### 9-2. 정합성은 사용자가 느끼는 신뢰와 연결된다

서버 내부 데이터가 맞더라도 사용자가 보는 화면에서 상태가 다르게 보이면 서비스 신뢰가 떨어질 수 있습니다.

그래서 백엔드 개발자는 DB 상태뿐 아니라 사용자가 보는 상태까지 고려해야 한다는 점을 배웠습니다.

---

### 9-3. 정책은 코드와 문서로 남겨야 한다

주문 상태, 실패 처리, 만료 처리, 마이페이지 노출 기준은 말로만 정하면 나중에 헷갈릴 수 있습니다.

정책을 코드, DTO, API 응답, 테스트, 문서로 남겨야 팀원이 같은 기준으로 유지보수할 수 있다는 점을 배웠습니다.

---

## 10. 한 줄 요약

On-Race에서 주문과 신청을 `Order.entryId`로 연결하고, signed prepare token과 confirm API로 결제 검증/확정 흐름을 정리해 신청, 주문, 결제, 마이페이지 상태가 서로 다르게 보이지 않도록 개선했습니다.
