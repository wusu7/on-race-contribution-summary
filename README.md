# On-Race Backend Contribution

> 팀 프로젝트 **On-Race**에서 제가 담당한 백엔드 기여 범위를 정리한 저장소입니다.  
> 전체 서비스 코드는 팀 협업 결과물이며, 이 저장소는 제가 맡은 **주문/결제 정합성, 마이페이지 API 계약, 주소 정책, 상태 노출 정책**을 중심으로 정리합니다.

---

## 1. 프로젝트 소개

**On-Race**는 사용자가 마라톤 참가 신청을 하고, 결제한 뒤, 마이페이지에서 신청/주문 상태를 확인하는 서비스입니다.

사용자 입장에서는 다음 흐름이 하나로 이어집니다.

```text
마라톤 신청 → 주문 생성 → 결제 진행 → 마이페이지에서 상태 확인
```

하지만 백엔드 내부에서는 신청, 주문, 결제, 마이페이지가 각각 다른 도메인으로 나뉘어 있습니다.

따라서 각 기능이 서로 다른 기준으로 동작하면 사용자는 실제 상태를 오해할 수 있습니다.

예를 들어:

```text
신청내역에는 참가 가능한 것처럼 보이는데
주문내역에는 결제 실패로 보이는 상황
```

이런 문제를 줄이기 위해 백엔드에서 주문과 신청을 연결하고, 결제 요청을 검증하며, 마이페이지에 보여줄 상태 기준을 정리했습니다.

## 5. 상세 기여

### 5-1. 주문/결제/신청 정합성 개선

#### 문제

결제 준비 단계와 실제 checkout 단계가 분리되어 있으면 다음과 같은 문제가 생길 수 있습니다.

```text
1. 사용자가 신청한 이벤트와 결제 요청의 이벤트가 달라질 수 있음
2. 사용자가 신청한 코스/페이스 정보와 결제 요청 정보가 달라질 수 있음
3. 이미 사용한 결제 준비 정보가 다시 사용될 수 있음
4. 주문이 생성된 뒤 어떤 신청에서 출발한 주문인지 추적하기 어려움
```

이 문제는 단순히 결제 API 하나의 문제가 아니라, 신청, 주문, 결제, 마이페이지 전체 흐름의 문제였습니다.

#### 해결

다음 작업을 통해 주문과 신청의 연결 기준을 정리했습니다.

- 주문 생성 시 `Order.entryId` 저장
- checkout 전 signed prepare token 검증 추가
- `POST /orders/{orderNumber}/confirm` API 추가
- 주문 확인과 마이페이지 노출 기준 정리
- 운영 SQL 및 런북 정리

#### 결과

주문이 어떤 신청에서 만들어졌는지 추적할 수 있게 되었고, 결제 요청이 올바른 신청 정보에서 출발했는지 검증할 수 있는 기준이 생겼습니다.

---

### 5-2. 마이페이지 API 계약 재구성

#### 문제

마이페이지는 단순히 데이터를 조회하는 화면이 아닙니다.

마이페이지에는 다음 정보가 함께 표시됩니다.

```text
계정 정보
주소 정보
신청 정보
주문 정보
결제 상태
```

이 정보들이 서로 다른 기준으로 내려오면 사용자는 자신의 실제 참가 상태를 이해하기 어렵습니다.

#### 해결

`/mypage/account` API를 재구성하고, 계정과 주소 도메인의 책임을 분리했습니다.

주요 작업은 다음과 같습니다.

- `/mypage/account` 집계 API 재구현
- auth와 address 책임 분리
- 기본 배송지 요약 DTO 정리
- 계정 상태값 정규화
- 관련 테스트 보강

#### 결과

프론트엔드가 마이페이지를 구성할 때 사용할 수 있는 응답 구조가 명확해졌고, 사용자에게 보여줄 계정/주소 정보의 의미가 안정화되었습니다.

---

### 5-3. 주소 정책 및 개인정보 로그 정비

#### 문제

주소 기능에서는 단순 CRUD보다 정책이 중요했습니다.

예를 들어 다음 상황을 고려해야 했습니다.

```text
기본배송지를 삭제하면 다음 기본배송지는 무엇인가?
기본배송지를 해제하면 아무 주소도 선택되지 않는가?
주문 전에 어떤 주소를 기본값으로 보여줄 것인가?
운영 로그에 개인정보가 남지는 않는가?
```

#### 해결

주소 정책을 다음과 같이 정리했습니다.

- 주소 CRUD 정책 정리
- 기본배송지 해제 시 최신 배송지 자동 승격
- DTO 통합
- PII 로그 제거
- 테스트 보강

#### 결과

주소 선택 흐름이 더 안정적으로 동작하게 되었고, 운영 로그에 개인정보가 남을 수 있는 위험을 줄였습니다.

---

### 5-4. 실패/만료 주문 노출 정책 정리

#### 문제

FAILED 또는 EXPIRED 상태의 주문이 신청내역에 다시 노출되면 사용자가 실제 참가 상태를 오해할 수 있습니다.

예를 들어:

```text
결제에 실패했는데 신청내역에는 아직 신청 가능한 것처럼 보이는 상황
```

이런 상태는 사용자 경험을 해칠 수 있기 때문에, 서버 상태와 화면 노출 정책을 함께 정리해야 했습니다.

#### 해결

- FAILED 주문 노출 정책 정리
- EXPIRED 주문 노출 정책 정리
- 신청내역 재노출 방지
- 취소 계열 상태 문구를 `결제취소`로 통일

#### 결과

사용자가 마이페이지에서 보는 상태와 서버의 주문 상태 사이의 혼란을 줄였습니다.

## 6. 기술적으로 배운 점

### 6-1. 결제 기능은 단순 API 호출이 아니다

결제 기능은 단순히 결제 요청을 보내는 기능이 아니었습니다.

신청 정보, 주문 정보, 결제 상태가 끝까지 같은 기준으로 유지되어야 했습니다.

그래서 결제 전 검증, 주문-신청 연결, 결제 완료 확정, 마이페이지 표시 정책이 함께 설계되어야 한다는 점을 배웠습니다.

---

### 6-2. 마이페이지 API는 화면 기준의 계약이다

마이페이지 API는 DB 데이터를 그대로 내려주는 API가 아니었습니다.

사용자가 이해할 수 있는 상태로 계정, 주소, 신청, 주문 정보를 정리해서 내려주는 계약이었습니다.

이 경험을 통해 백엔드 API는 프론트엔드 화면 요구사항과 사용자 경험을 함께 고려해야 한다는 점을 배웠습니다.

---

### 6-3. 도메인 정책은 코드와 테스트로 고정해야 한다

기본배송지, 주문 상태, 실패/만료 주문 노출처럼 서비스 규칙은 말로만 정하면 유지되기 어렵습니다.

정책을 코드, DTO, API 응답, 테스트로 고정해야 팀원이 같은 기준으로 이해하고 유지보수할 수 있다는 점을 배웠습니다.

- AWS EC2 환경 정보
- Nginx 설정 확인 및 reload 명령
- 로그 확인 명령

운영 문서화는 기능 구현 이후에도 팀원이 같은 환경에서 문제를 확인하고 대응할 수 있도록 만드는 작업이라고 이해했습니다.
